# 80 - Agentic 执行模式

## 定位

本模块定义 Agent 在任务契约内如何更稳定地执行工作。

它补充 `20-governance.md`，但不替代任务等级、授权等级、验收门或 vault 记忆系统。治理层回答“Agent 被授权做什么、怎样关闭任务”；执行模式回答“Agent 如何把任务做得可接力、可验证、可复利”。

## 来源边界

本模块吸收 `support.md` 中可迁移的 Agentic Engineering 经验：

- 计划先行；
- 上下文扎根；
- 检查点化工作；
- 人类提供判断信号；
- 重复工作沉淀为 skill。

不吸收个人工具栈、权限绕过设置、远程控制配置、邮箱 daemon、生活服务 CLI 或任何真实凭据配置。那些可以作为个人实践参考，不属于默认初始化协议。

## 执行原则

### Plan First

除明显一行改动外，Agent 在实施前应先形成轻量计划或任务契约。

计划不需要总是长文档，但必须回答：

- 当前目标是什么；
- 成功如何验证；
- 需要读取哪些上下文；
- 预计触碰哪些文件或边界；
- 哪些内容明确不做。

Level B 和 Level C 任务必须使用任务文件承载计划、验收和执行记录。

### Context Grounded

Agent 的方案必须基于当前项目上下文，而不是泛化建议。

执行前优先读取：

- Agent 入口文件；
- `vault/index.md` 路由到的上下文；
- `vault/runtime.md` 当前状态；
- `vault/governance.md` 任务规则；
- 相关任务文件、决策记录和本地代码模式。

如果需要使用外部资料、社区信息或第三方文档，必须明确其用途和不确定性。单元测试不得真实调用外部服务。

### Checkpointable Work

长任务应能跨会话恢复。

可恢复检查点包括：

- `vault/tasks/*` 中的任务契约和执行记录；
- `vault/runtime.md` 中的活跃任务指针和当前状态；
- `vault/handoff.md` 中的接力信息；
- 明确的验收标准和验证命令。

当上下文窗口、工具会话或 Agent 发生切换时，后续 Agent 应能通过这些文件继续，而不是依赖对话记忆。

### Human Signal

Agent 负责执行量、搜索、整理、实现和验证；人类负责方向、取舍、品味、风险接受和最终验收。

当出现以下情况时，Agent 应把问题显式交还给人类判断：

- 存在多个合理方向且影响后续成本；
- 任务会改变架构、公开接口、数据模型、安全、权限、隐私、成本或部署行为；
- 验收标准本身不清晰；
- 继续推进需要扩大授权范围。

Agent 不应把“我能做到”误认为“这是应该做的”。

### Workflow Compounding

重复出现的工作流应沉淀为 skill，而不是长期塞入口文件或临时提示词。

适合沉淀为 skill 的信号：

- 同类任务已重复两次以上；
- 执行步骤稳定；
- 检查清单可复用；
- 失败模式可预防；
- 多个项目或多个 Agent 会复用。

新增 skill 应保持聚焦，并在 `40-skills.md` 的边界内演进。不要在读取路径尚未稳定前创建大量 skill。

## 与治理层的关系

- Level A 任务可以使用轻量计划，并在 `vault/runtime.md` 记录必要状态。
- Level B 任务必须把计划、执行记录和验证结果写入任务文件。
- Level C 任务必须记录长期决策，并在需要时先获得用户确认。
- 任何执行模式都不能降低 `20-governance.md` 中的授权和禁止规则。

## 反模式

避免以下做法：

- 没读项目上下文就直接套用通用方案；
- 用长入口文件替代 vault、任务文件或 skill；
- 因为计划存在就跳过验收；
- 因为测试通过就宣布任务完成；
- 把个人机器配置、权限绕过或外部服务账号作为项目默认要求；
- 为一次性任务创建复杂框架或大量 skill。
